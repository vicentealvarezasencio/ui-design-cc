# V0 Adapter

Rules for generating Vercel V0-optimized prompts from UI specifications.

<adapter_info>
Service: V0 by Vercel
URL: https://v0.dev/
Output: React components with Tailwind CSS and shadcn/ui
Strength: Production-ready React code, component-level generation
</adapter_info>

<prompt_structure>

V0 prompts should follow this structure:

```
Create a [component/page] using shadcn/ui components:

- [Structure with component names]
- [Props and variants]
- [Styling with Tailwind classes]

[Behavior notes if interactive]
[Responsive notes]
```

</prompt_structure>

<transformation_rules>

## Screen Spec → V0 Prompt

1. **Reference shadcn/ui:**
   - V0 is trained on shadcn/ui
   - Use exact component names: Button, Input, Card, etc.
   - Reference variants: variant="outline", size="lg"

2. **Use Tailwind terminology:**
   - Classes like max-w-md, mx-auto, space-y-4
   - Color utilities: bg-primary, text-muted-foreground
   - Spacing: p-6, gap-4, mb-2

3. **Structure clearly:**
   - Describe nesting (Card containing Form containing Input)
   - Use indentation in descriptions
   - Specify flex/grid layouts

4. **Include interactivity:**
   - Form submissions
   - Click handlers
   - State changes

</transformation_rules>

<token_mapping>

V0/shadcn uses CSS variables. Map tokens to Tailwind:

| Token Reference | V0/Tailwind |
|-----------------|-------------|
| color.primary.default | bg-primary, text-primary |
| color.text.muted | text-muted-foreground |
| color.background.default | bg-background |
| color.border.default | border-border |
| spacing.4 | p-4, m-4, gap-4 |
| border.radius.md | rounded-md |
| shadow.md | shadow-md |

</token_mapping>

<example_transformation>

**Input: Screen Spec**
```markdown
# SCR-01: Login

Route: /login
Layout: centered-card (max-width: 400px)

## Components
1. Logo — centered, 48px height
2. Heading — "Welcome back", typography.h2
3. InputField — label: "Email", type: email
4. InputField — label: "Password", type: password
5. Button — label: "Sign In", variant: primary, fullWidth
6. Link — "Forgot password?", centered
7. Divider — text: "or"
8. Button — label: "Continue with Google", variant: secondary, icon: google
```

**Output: V0 Prompt**
```
Create a login page using shadcn/ui components:

- Card centered on page with max-w-md and mx-auto
- Inside Card:
  - CardHeader:
    - Image placeholder for logo (h-12, mx-auto, mb-4)
    - CardTitle "Welcome back" centered
  - CardContent with space-y-4:
    - Form with:
      - div with Label "Email" and Input type="email"
      - div with Label "Password" and Input type="password"
      - Button type="submit" className="w-full">Sign In</Button>
    - Link "Forgot password?" centered below form, text-sm text-muted-foreground
    - Separator with text "or" in the middle
    - Button variant="outline" className="w-full" with Google icon on left: "Continue with Google"

Use Tailwind for styling. Primary button uses default variant.
Form should have proper spacing between elements (space-y-4).
```

</example_transformation>

<advanced_patterns>

**Forms with validation:**
```
Create a signup form using shadcn/ui Form component with react-hook-form:
- Email field with email validation
- Password field with minimum length validation
- Show inline error messages below fields
- Disable submit button while submitting
```

**Data tables:**
```
Create a data table using shadcn/ui Table with:
- Sortable columns for Name, Date, Status
- Pagination at bottom
- Search/filter input above
- Row actions dropdown (Edit, Delete)
```

**Dashboard layouts:**
```
Create a dashboard layout with:
- Sidebar using shadcn/ui navigation
- Top header with user dropdown
- Main content area with Card grid
- Responsive: sidebar collapses on mobile
```

</advanced_patterns>

<best_practices>

**Do:**
- Reference shadcn/ui component names exactly
- Use Tailwind class names for styling
- Describe component hierarchy clearly
- Mention responsive behavior
- Include form validation requirements

**Don't:**
- Use generic terms like "input box" (say "Input")
- Skip the component library reference
- Forget to specify variants
- Ignore mobile considerations

</best_practices>
